# Slither Static Analysis Notes

This document explains the Slither findings for reviewers and auditors.

## Summary

Slither v0.11.4 was run against `packages/contracts/src/` on April 9, 2026.
102 results found across 23 contracts. The majority are false positives or
expected patterns in ZK protocol architecture.

## Category 1: False Positives & ZK-Generated Code (No Action Required)

**`assembly`, `incorrect-return`, `solc-version`, `pragma`, `naming-convention`**
All findings point to `Groth16VerifierCore.sol`, which is auto-generated by
SnarkJS. Inline assembly is required for efficient BN254 pairing verification
on-chain and is the industry standard for Groth16 verifiers. No action required.

**`incorrect-equality` (chainId == block.chainid in AttestationVerifier)**
This strict equality is intentional. It implements cross-chain replay protection
as documented in `docs/PUBLIC_SECURITY_MODEL.md`. This is a security feature,
not a bug.

## Category 2: Safe by Architecture (No Action Required)

**`reentrancy-no-eth` / `reentrancy-benign` / `reentrancy-events`**
Slither flags external calls to `_nullifierRegistry.spend()` before state
transitions. In `TransferVault.execute()`, the state transition
`_state = VaultState.EXECUTED` already occurs before `_creditAssets()` â€”
the CEI pattern is correctly implemented. The `_nullifierRegistry` is a
trusted internal contract. No reentrancy vector exists.

**`timestamp`**
`block.timestamp` is used for the 24-hour governance timelock and the
proof-of-life challenge window. This is standard practice for time-gated
on-chain logic and is intentional by design.

**`low-level-calls`**
Low-level calls in `AfterchainGovernance.executeAction()` and
`TransferVault.withdrawETH()` are guarded by governance controls and
pull-based withdrawal patterns respectively.

**`calls-loop`**
External ERC-20 `balanceOf` calls inside asset loops are bounded by the
registered asset list, which is controlled by the vault owner at setup time.

## Category 3: Resolved

**`divide-before-multiply` (TransferVault._computeCuts)**
Fixed. Fee calculation in `_computeCuts` was refactored to eliminate
intermediate division before multiplication:

- Before: `(assetFee * 20) / 100`
- After: `(balance * fb.feeEurCents * 20) / (fb.walletValue * 100)`

Committed: `fix: resolve divide-before-multiply precision loss in _computeCuts`

**`missing-zero-check`**
Fixed. Zero-address validation added to:
- `TransferVault.constructor` â€” `owner_` parameter
- `NullifierRegistry.setOperator` â€” `operator_` parameter
- `TransferVaultFactory.constructor` â€” `multiSigVerifier_` and `feeTermsVerifier_` parameters

Committed: `fix: add zero-address checks and immutable declarations per Slither recommendations`

**`immutable-states`**
Fixed. The following state variables are now declared `immutable`:
- `TransferVault.owner`
- `TransferVault.templateId`
- `TransferVault._challengeWindowDuration`

Committed: `fix: add zero-address checks and immutable declarations per Slither recommendations`

**Test harness updates**
Four test files were updated to use placeholder addresses (address(0xDEAD1), address(0xDEAD2)) instead of address(0) for disabled features, since the factory now enforces non-zero addresses. The test_attestMultiSig_notConfigured_reverts test was refactored to construct TransferVault directly, bypassing the factory, to preserve coverage of the zero-multiSigVerifier code path.

**`cache-array-length`**
Noted. Gas optimization scheduled for pre-audit hardening pass.
